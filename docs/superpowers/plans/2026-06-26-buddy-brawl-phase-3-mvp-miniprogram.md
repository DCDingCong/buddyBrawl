# Buddy Brawl 三阶段首版小程序计划

> 面向智能体执行：三阶段优先使用小步验收，不建议同时大改后端和小程序。除非发现阻塞问题，否则后端只做必要补缺，小程序体验收口优先。

## 1. 阶段目标

三阶段完成后，尽可能交付一个用户可试用的首版本微信小程序。

首版用户应能完成：

1. 进入小程序。
2. 通过微信手机号授权自动进入账号流程。
3. 如果玩家数据完整，直接进入自己的熊猫主页。
4. 如果玩家数据不完整，进入角色注册/初始化流程。
5. 查看熊猫体型、状态、局部装饰和花纹。
6. 领取挂机收益。
7. 查看装备并使用金币强化。
8. 发起异步竞技战斗。
9. 查看战斗结果和战报。
10. 查看排行榜。

本阶段不追求完整商业化上线，但要从“调试页面”提升到“可给用户体验的最小产品”。

## 2. 阶段原则

- 用户可见文案全部中文。
- 页面围绕玩法目标，不展示工程调试感。
- 未登录状态不展示传统登录页，进入小程序后通过微信手机号授权弹窗自动进入账号流程。
- 手机号授权、玩家初始化、熊猫角色注册必须拆成独立模块，不能互相写死。
- 后端接口尽量稳定，不在三阶段大幅重构。
- 不新增复杂系统，优先打磨已有闭环。
- 装备强化首版只消耗金币，但强化配置和扣费接口必须保留特殊道具、特殊游戏货币、充值货币的扩展位。
- 熊猫角色只允许换局部装饰或花纹，不做整套角色外观随意替换。
- 错误状态必须可理解，不能只显示英文异常。
- 所有主要流程必须能在微信开发者工具中手工跑通。

## 3. 开发范围

### 3.1 微信手机号授权自动登录

目标：

- 用户进入小程序后，不展示传统登录页。
- 小程序调用微信手机号授权弹窗。
- 后端根据手机号查找或创建账号。
- 后端判断玩家数据是否完整，并返回下一步去向。

推荐流程：

1. 小程序启动。
2. 首页检测本地登录态。
3. 没有登录态时显示“开始游戏”主按钮。
4. 点击后触发微信手机号授权。
5. 小程序把微信返回的手机号授权凭证交给服务端。
6. 服务端完成手机号校验、账号查找或账号创建。
7. 如果玩家数据完整，返回 `nextAction: "enter_game"`。
8. 如果玩家数据不完整，返回 `nextAction: "complete_profile"`。
9. 小程序分别进入主页或角色注册/初始化页。

需要新增或调整：

- `apps/miniprogram/services/auth.ts`
- `apps/miniprogram/pages/home/*`
- `apps/miniprogram/pages/profile-setup/*`
- `services/api/src/routes/auth.ts`
- `services/api/src/modules/auth/*`
- `services/api/src/modules/player/*`
- `packages/shared/src/api-types.ts`

验收标准：

- 未登录时不需要用户输入账号密码。
- 用户授权手机号后，已存在完整玩家数据时直接进入主页。
- 用户授权手机号后，缺少完整玩家数据时进入注册/角色初始化页。
- 登录态和玩家初始化状态由服务端判断，前端不硬猜。
- 保留本地开发登录能力，但只能作为开发环境后门，不作为首版用户路径。

### 3.2 熊猫角色体型与局部外观系统

目标：

- 熊猫角色参考《光遇》的差异感：每个玩家拥有不同的初始身材值。
- 初始身材值用于形成角色辨识度，不作为频繁更换的商品化内容。
- 玩家后续只能更换局部装饰或花纹，避免角色系统和装备系统混在一起。

数据模型建议：

- `bodyProfile`：身高、体型、头身比、基础轮廓、体态标签。
- `appearanceSlots`：头饰、脸部花纹、身体花纹、背饰、手持装饰。
- `cosmeticInventory`：玩家拥有的外观装饰。
- `equipmentInstances`：战斗装备，继续只负责属性和战斗。

模块边界：

- 熊猫体型模块只负责初始身材生成和展示参数。
- 外观装饰模块只负责局部装饰、花纹和展示组合。
- 装备模块只负责战斗装备、属性、穿戴和强化。
- 战斗引擎只读取装备和宠物属性，不直接读取外观装饰。

验收标准：

- 新玩家初始化时生成一组稳定的熊猫体型参数。
- 同一个玩家重复登录不会重新随机体型。
- 首页能展示熊猫体型和当前局部装饰摘要。
- 更换局部装饰或花纹不影响战斗属性。
- 后续增加新装饰槽时，不需要改战斗引擎。

### 3.3 装备强化金币化与货币扩展

目标：

- 首版装备强化只开放金币升级。
- 不再把强化材料作为首版必需消耗。
- 保留后续特殊道具、特殊游戏货币和充值货币的扩展接口。

强化配置建议：

```ts
interface EnhanceCostConfig {
  gold: number;
  requiredItems: Array<{
    itemConfigId: string;
    amount: number;
  }>;
  specialCurrency: Array<{
    currencyId: string;
    amount: number;
  }>;
  paidCurrency: Array<{
    currencyId: string;
    amount: number;
  }>;
}
```

首版配置规则：

- `gold` 大于 0。
- `requiredItems` 为空数组。
- `specialCurrency` 为空数组。
- `paidCurrency` 为空数组。

扩展预留：

- 后续如果加入类似“金豆”的特殊游戏货币，只增加配置和钱包扣费策略。
- 后续如果加入充值货币，只增加充值货币账户和扣费策略，强化服务仍读取同一份成本配置。
- 首版不做充值、不做商城、不做付费强化。

模块边界：

- 强化模块负责读取强化配置、校验等级上限、发起扣费、提升强化等级。
- 钱包模块负责金币、特殊货币、充值货币的余额读取和扣减。
- 道具模块负责后续道具库存，不和金币余额混在同一逻辑里。
- 装备模块不直接判断“充值货币”业务，只消费统一扣费接口的结果。

验收标准：

- 首版强化只扣金币。
- 金币不足时返回中文错误提示。
- 强化成功后装备等级提升，金币减少。
- 配置结构中保留道具、特殊货币、充值货币字段。
- 首版不出现充值入口。

### 3.4 小程序全局中文化和基础包装

目标：

- 把当前小程序从英文调试界面改为中文首版界面。
- 建立原创熊猫乱斗的基础氛围。

需要修改：

- `apps/miniprogram/app.json`
- `apps/miniprogram/app.wxss`
- `apps/miniprogram/pages/home/*`
- `apps/miniprogram/pages/adventure/*`
- `apps/miniprogram/pages/equipment/*`
- `apps/miniprogram/pages/arena/*`
- `apps/miniprogram/pages/battle-report/*`
- `apps/miniprogram/services/api.ts`

验收标准：

- tab 文案为中文：主页、冒险、装备、竞技。
- 页面标题为中文。
- 按钮、空状态、错误提示为中文。
- 首页第一屏能明确表达“我的熊猫”和“下一步操作”。

### 3.5 首页体验收口

目标：

- 首页成为玩家进入游戏后的主控台。
- 玩家能一眼看到熊猫体型状态、局部装饰、资源、挂机收益、竞技积分和推荐行动。

建议内容：

- 玩家昵称。
- 熊猫名称、体型标签、等级、经验、核心属性。
- 当前局部装饰和花纹摘要。
- 金币。
- 当前挂机关卡和可领取收益。
- 当前竞技积分。
- 今日可做事项：领取收益、强化装备、挑战竞技。

需要补充：

- 未登录时显示“开始游戏”按钮，点击后触发微信手机号授权。
- 手机号授权完成后，根据服务端 `nextAction` 进入主页或角色注册/初始化页。
- 首页提供跳转到挂机、装备、竞技的明确按钮。

验收标准：

- 新用户第一次进入可以通过一个主按钮触发手机号授权。
- 登录成功后不用手动理解 token、手机号授权凭证或 API 地址。
- 首页不显示工程字段名，例如 `token`、`devOpenId`、`battleId` 等。

### 3.6 角色注册/初始化页

目标：

- 当手机号对应账号没有完整玩家数据时，引导用户完成首个熊猫角色初始化。
- 注册页只处理角色初始化，不处理手机号认证。

建议内容：

- 熊猫默认名称，可修改。
- 初始体型预览。
- 初始花纹或局部装饰预览。
- “进入竹林”或类似中文确认按钮。

验收标准：

- 用户授权手机号但没有完整玩家数据时进入该页。
- 确认后创建完整玩家数据、熊猫体型、初始外观、初始装备和初始资源。
- 重复提交不会创建多个主宠物。
- 初始化完成后进入主页。

### 3.7 挂机冒险页面收口

目标：

- 玩家能理解挂机收益来自哪里，并能领取。
- 玩家能进行一次关卡自动战斗。

需要优化：

- 把 `gold`、`exp` 显示为金币、经验。
- 如果接口仍返回 `enhanceMaterial`，首版只作为后续扩展材料或预留资源展示，不作为装备强化必需消耗。
- 把 `attacker`、`defender`、`winner` 等战斗字段转成中文。
- 战斗结果显示胜负、奖励和关键回合。

验收标准：

- 页面打开后自动加载挂机状态。
- 领取收益后页面刷新资源和收益状态。
- 关卡战斗后显示中文战斗结果。

### 3.8 装备页面收口

目标：

- 玩家能理解每件装备的部位、等级、属性和可操作状态。

需要优化：

- 部位中文化：武器、头部、身体、饰品。
- 品质中文化：普通、精良、稀有、史诗。
- 装备状态中文化：已穿戴、可穿戴、可强化、满级。
- 强化失败时给出中文原因，例如材料不足、已满级。

验收标准：

- 玩家能完成穿戴和金币强化。
- 强化后装备等级、金币数量、首页属性展示有变化。
- 不出现英文按钮或英文空状态。
- 页面不出现强化材料作为首版必需消耗。

### 3.9 竞技和战报页面收口

目标：

- 玩家能挑战其他真实玩家，并理解战斗结果。

需要优化：

- 对手列表展示昵称、宠物、等级、积分。
- 挑战按钮中文化。
- 战斗结果展示胜负、积分变化、战斗摘要。
- 最近战报列表可进入详情页。
- 战报详情按回合展示中文战斗文本。

验收标准：

- 至少两个玩家存在时，可以发起竞技挑战。
- 挑战后生成战报。
- 战报详情不重新结算，只展示已保存内容。

### 3.10 排行榜入口

目标：

- 用户能看到排行榜和自己的排名。

实现方式：

- 可以在竞技页内加入排行榜区块。
- 不一定单独新建 tab。

验收标准：

- 展示前若干名玩家。
- 展示我的排名。
- 排名和竞技积分变化一致。

### 3.11 任务和新手引导

目标：

- 给用户一个最小的新手目标链。

建议目标链：

1. 领取一次挂机收益。
2. 强化一次装备。
3. 进行一次竞技挑战。
4. 查看一次战报。

实现方式：

- 首页展示“今日目标”。
- 任务可领取时显示明确按钮。
- 领取后展示奖励变化。

验收标准：

- 用户不用阅读说明也能知道下一步做什么。
- 任务状态和后端 `/tasks` 保持一致。

### 3.12 后端必要补缺

原则：

- 不做大范围重构。
- 只补小程序首版需要的缺口。

可能需要补充：

- 微信手机号授权登录接口。
- 玩家注册状态判断。
- 熊猫体型参数和局部外观槽位。
- 金币强化配置和钱包扣费接口。
- 统一错误码到中文提示映射。
- 首页返回资源字段，减少小程序多次请求。
- 战斗结果返回积分变化。
- 排行榜返回我的排名时包含排名数字。
- 开发环境种子数据入口，便于创建多个对手。

验收标准：

- 每个补缺都有 API 测试。
- `pnpm test` 和 `pnpm typecheck` 通过。

## 4. 模块化边界要求

三阶段实现必须按模块边界开发，避免后续调整导致大范围重写。

### 4.1 账号认证模块

职责：

- 接收微信手机号授权结果。
- 换取和校验手机号。
- 查找或创建账号。
- 签发服务端登录态。

不负责：

- 创建熊猫角色。
- 创建装备。
- 决定战斗数据。

### 4.2 玩家初始化模块

职责：

- 判断玩家数据是否完整。
- 创建玩家档案。
- 创建默认熊猫。
- 创建初始资源、初始装备、挂机状态、竞技状态和任务进度。

不负责：

- 微信手机号校验。
- 装备强化规则。
- 外观装饰穿戴细节。

### 4.3 熊猫体型模块

职责：

- 生成稳定的初始体型值。
- 提供体型展示字段。
- 保证重复登录不重置体型。

不负责：

- 局部装饰库存。
- 战斗属性计算。
- 装备强化。

### 4.4 外观装饰模块

职责：

- 管理局部装饰、花纹和展示槽位。
- 支持后续新增装饰槽。

不负责：

- 战斗属性。
- 装备属性。
- 付费逻辑。

### 4.5 钱包模块

职责：

- 管理金币余额。
- 预留特殊游戏货币和充值货币账户。
- 提供统一扣费接口。

不负责：

- 决定装备强化上限。
- 决定任务完成条件。
- 决定充值商品。

### 4.6 装备强化模块

职责：

- 读取强化成本配置。
- 校验强化等级上限。
- 调用钱包模块扣费。
- 提升装备强化等级。

不负责：

- 直接操作手机号登录。
- 直接判断充值货币业务。
- 管理外观装饰。

## 5. 三阶段调试计划

### 5.1 自动化验证

每轮后端修改后运行：

```bash
pnpm test
pnpm typecheck
```

涉及 Docker 后运行：

```bash
docker compose up -d --build api
docker compose exec -T api pnpm prisma migrate deploy
curl http://127.0.0.1:3000/health
```

### 5.2 小程序手工验收

在微信开发者工具中检查：

- 项目目录为 `apps/miniprogram`。
- 已开启“不校验合法域名”。
- 本地 API 地址为 `http://127.0.0.1:3000`。
- 首页可进入游戏。
- 未登录时可以触发微信手机号授权。
- 授权后能根据玩家数据完整度进入主页或角色注册/初始化页。
- tab 可正常切换。
- 挂机、装备、竞技、战报页面不报错。
- 控制台无明显接口异常。

### 5.3 用户路径验收

按真实用户路径执行：

1. 打开小程序。
2. 点击“开始游戏”，触发微信手机号授权。
3. 如果是新用户，完成熊猫角色初始化。
4. 查看熊猫体型、局部装饰和状态。
5. 领取挂机收益。
6. 去装备页用金币强化一件装备。
7. 去竞技页挑战一个对手。
8. 查看战报。
9. 回到竞技页查看排行榜。

通过标准：

- 全流程不需要手动复制 token。
- 页面不出现英文主文案。
- 失败时有中文提示。
- 用户能明确知道自己获得了什么、提升了什么、打赢或打输了谁。

## 6. 三阶段不纳入范围

以下内容不阻塞首版试用：

- 支付和广告。
- 好友挑战。
- 赛季系统。
- 配置后台。
- 完整皮肤、套装和武器美术系统。
- 充值货币实际购买和消耗。
- 特殊游戏货币正式产出系统。
- 外观装饰商城。
- 正式线上部署。
- 性能压测。

## 7. 阶段完成定义

三阶段完成时应满足：

- 小程序可在微信开发者工具中完整跑通首版用户路径。
- 微信手机号授权自动登录路径可用。
- 账号完整时进入主页，账号不完整时进入角色注册/初始化页。
- 新玩家拥有稳定的熊猫体型值。
- 熊猫只支持局部装饰和花纹变化，不支持整套角色随意替换。
- 装备强化首版只消耗金币。
- 强化配置保留道具、特殊货币、充值货币扩展字段。
- 用户可见文案中文化。
- 后端接口保持二阶段闭环稳定。
- 自动化测试通过。
- Docker 本地环境可启动。
